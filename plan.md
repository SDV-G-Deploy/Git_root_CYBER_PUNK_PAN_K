# One Shot Lab - MVP Plan

## 1) Цель MVP (что считается играбельным)
- Игрок проходит один короткий уровень-арену, делая по одному выстрелу за ход.
- Выстрел запускает физическое взаимодействие и потенциальную цепную реакцию.
- На экране есть предиктор траектории (прицеливание), объекты реакции и счет.
- Раунд завершается, когда все цели уничтожены или исчерпаны ходы.
- Результат отображается как `Score`, `Chains`, `Turns Used`, `Retry`.

Играбельный MVP = можно открыть страницу в браузере, сыграть полный раунд от старта до финального экрана без ошибок в консоли.

## 2) Scope In / Scope Out

### Scope In
- 1 арена (фиксированный layout).
- 3 типа объектов:
  - `Node` (обычная цель, дает очки).
  - `Trigger` (активирует соседние узлы/события).
  - `Block` (препятствие, влияет на траекторию).
- 1 снаряд на ход.
- Подсветка траектории перед выстрелом.
- Базовая цепная реакция по правилам близости/контакта.
- Подсчет очков и экран результата.
- Кнопка `Retry`.

### Scope Out
- Мультиуровни, редактор уровней.
- Сложная физика (вращения, материалы, soft-body).
- Мобильные жесты с продвинутой haptics.
- Анимации высокого продакшн-качества и VFX-пакеты.
- Аудиосистема с микшированием.
- Аккаунты, облачные сохранения, лидерборды.

## 3) Архитектура файлов (какие файлы и за что отвечают)
- `index.html` - корневой HTML, canvas и UI-контейнеры.
- `styles.css` - базовая сетка, HUD, экран финала.
- `src/main.js` - bootstrap, запуск game loop.
- `src/config.js` - константы (размеры арены, скорость, scoring, debug flags).
- `src/state.js` - фабрика и обновление `gameState`.
- `src/input.js` - мышь/тач: aim, hold, release.
- `src/simulate.js` - шаг симуляции (полет снаряда, столкновения, таймеры).
- `src/resolve.js` - разрешение цепочек и применение эффектов.
- `src/score.js` - расчет очков, бонусов за цепь.
- `src/entities/projectile.js` - логика снаряда.
- `src/entities/node.js` - поведение цели.
- `src/entities/trigger.js` - поведение триггера.
- `src/entities/chain.js` - сборка и хранение цепочки событий.
- `src/levels/level1.js` - данные единственной арены.
- `src/ui.js` - HUD, подсказки, финальный экран.
- `src/telemetry.js` - события и буфер логов.

## 4) Игровой цикл (input -> simulate -> resolve -> score -> next turn)
1. `input`:
   - Игрок двигает указатель, система считает вектор выстрела.
   - Отображается предиктор траектории.
   - На `release` фиксируется shot.
2. `simulate`:
   - Создается `Projectile`.
   - По тикам обновляются позиция/скорость.
   - Проверяются столкновения с `Node/Trigger/Block`.
3. `resolve`:
   - Попавшие объекты переводятся в `activated/resolved`.
   - Триггеры распространяют эффект на соседние объекты.
   - Формируется `Chain` (order, depth, size).
4. `score`:
   - Базовые очки за объект + множитель за глубину/длину цепи.
   - Обновляется HUD.
5. `next turn`:
   - Если цели еще есть и лимит ходов не исчерпан -> следующий ход.
   - Иначе `level_complete` или `run_failed` и показ финального UI.

## 6) Основные сущности (Projectile, Node, Trigger, Chain, Level, UI)
- `Projectile`:
  - Поля: position, velocity, radius, alive, ttl.
  - Ответственность: движение, коллизии, завершение полета.
- `Node`:
  - Поля: id, hp(=1), position, radius, state.
  - Ответственность: принимать урон, давать очки, участвовать в цепи.
- `Trigger`:
  - Поля: id, position, radius, cooldown, state.
  - Ответственность: при активации воздействовать на соседние сущности.
- `Chain`:
  - Поля: links[], depth, startedAtTick, resolvedAtTick.
  - Ответственность: порядок реакций, множитель очков, событие резолва.
- `Level`:
  - Поля: id, bounds, spawn, entities, winCondition, turnLimit.
  - Ответственность: декларативная конфигурация арены.
- `UI`:
  - Поля: hudState, endScreenState.
  - Ответственность: рендер счета/ходов/подсказок, retry flow.

## 7) События телеметрии (run_start, shot_fired, chain_resolved, level_complete, retry)
- `run_start`
  - Когда: при старте/рестарте уровня.
  - Payload: `{ runId, levelId, ts }`.
- `shot_fired`
  - Когда: отпускание выстрела.
  - Payload: `{ runId, turn, origin, aimVector, power, ts }`.
- `chain_resolved`
  - Когда: после завершения цепной реакции.
  - Payload: `{ runId, turn, linksCount, depth, scoreDelta, ts }`.
- `level_complete`
  - Когда: выполнен win condition.
  - Payload: `{ runId, levelId, turnsUsed, finalScore, ts }`.
- `retry`
  - Когда: игрок нажал перезапуск.
  - Payload: `{ previousRunId, levelId, ts }`.

Принцип: сначала логируем в памяти (`telemetryBuffer`), позже можно добавить отправку на backend.

## 8) Критерии готовности (DoD)
- Проект запускается локально открытием `index.html` или через простой static server.
- Полный цикл хода работает без runtime-ошибок.
- Предиктор траектории визуально совпадает с фактическим стартом полета.
- Все 3 типа объектов присутствуют и имеют различимое поведение.
- Цепочка корректно собирается и влияет на счет.
- Есть финальное состояние `success/fail` и кнопка `Retry`.
- Телеметрия 5 событий пишется в консоль/буфер.
- Код разделен по файлам из архитектуры, без монолитного `main.js`.

## 9) Риски и анти-паттерны
- Риск: смешивание рендера и логики в одном модуле.
  - Анти-паттерн: "god-file" на 1000+ строк.
  - Митигирование: жесткое разделение `input/simulate/resolve/ui`.
- Риск: нестабильная физика из-за большого `deltaTime`.
  - Анти-паттерн: frame-dependent симуляция.
  - Митигирование: fixed timestep (например 1/60).
- Риск: цепочка с рекурсией без лимитов.
  - Анти-паттерн: бесконечные триггеры.
  - Митигирование: visited-set + maxDepth.
- Риск: невалидный state после retry.
  - Анти-паттерн: частичный reset.
  - Митигирование: полная реинициализация через `createInitialState(level)`.
- Риск: преждевременная оптимизация.
  - Анти-паттерн: сложные абстракции до появления боли.
  - Митигирование: сначала читаемость, потом профилирование.
